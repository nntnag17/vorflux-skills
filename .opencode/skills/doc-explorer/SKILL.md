# doc-explorer — Code-First Documentation Agent

Read the actual source code, understand what it does, and write a clear, accurate README that a developer can use immediately. No generic placeholders. No guessing. Every claim in the README is backed by something real in the codebase.

**Difference from `readme-gen`:** `readme-gen` fills a template from file-level metadata (package.json, go.mod). `doc-explorer` reads entry points, extracts real function signatures, scrapes routes and CLI flags, and pulls usage examples from test files — then writes documentation grounded in what the code actually does.

## How to use

```
/doc-explorer [directory or file]
```

## Instructions

### Step 1 — Run the codebase explorer

Scan the project structure and extract real metadata:

```bash
python3 ${SKILL_DIR}/scripts/explore_codebase.py "$ARGUMENTS"
```

If the argument is a file, the script analyses its containing directory.

This outputs a JSON report with:
- Detected language, framework, and runtime
- Entry points (main files, CLI scripts, server start commands)
- Public API surface (exported functions, classes, routes)
- Environment variables and config keys
- Dependencies and their purposes
- Existing docs (LICENSE, CONTRIBUTING, CHANGELOG)

### Step 2 — Read the entry points

After running the script, read the actual entry point files identified in the JSON. For each entry point:

- Read the file fully if it is under 300 lines
- For larger files, read the top 80 lines (imports, class/function definitions) and the bottom 20 lines

Look for:
- What the program does when it starts
- What the main exported functions/classes are and what they accept/return
- Any inline comments or docstrings that explain intent
- Error handling patterns that hint at failure modes

### Step 3 — Read supporting files

Read up to 5 additional files that reveal how the project is used:

- Test files (they show real usage examples)
- Example scripts or `examples/` directory
- Config files (reveal required setup)
- CLI entry points (`bin/`, `cmd/`, `__main__.py`)

### Step 4 — Synthesize understanding

Before writing, answer these questions from what you read:

1. **What does this project do in one sentence?**
2. **Who is the intended user?** (developer, end user, ops engineer)
3. **What is the primary interface?** (CLI, library, HTTP API, UI)
4. **What are the 3–5 most important things a new user needs to know?**
5. **What can go wrong and how does the user fix it?**

Do not proceed to writing until you can answer all five questions with specifics from the code.

### Step 5 — Write the README

Use `templates/README_sections.md` as the section guide. Write each section using only information you confirmed in Steps 2–4.

**Rules:**
- Every code example must be real and runnable (copy-paste from actual source or tests)
- Every environment variable must be one you actually found in the code
- Every feature bullet must describe something the code actually does
- No `TODO`, `your-project`, `example.com`, or other placeholder text
- Use concrete names: actual function names, real flag names, real config keys

**Output file:**
- If no `README.md` exists: write to `README.md`
- If `README.md` already exists: write to `README_generated.md`. In your final report to the user, list which sections are new vs. already present in the original, so they can decide what to merge.

### Step 6 — Self-check

Before finishing, verify:

- [ ] Every code block has a language hint (` ```bash `, ` ```python `, ` ```go `, etc.)
- [ ] The installation section produces a working setup when followed exactly
- [ ] No section references a file or command that does not exist in the repo
- [ ] The one-line description in the title matches what the code actually does
- [ ] All links are relative paths that exist, or external URLs that look valid

### Step 7 — Report to user

Tell the user:
- Where the README was written
- The 3 most important things you learned about the codebase that shaped the docs
- Any gaps you found (missing LICENSE, no .env.example, undocumented required config)
